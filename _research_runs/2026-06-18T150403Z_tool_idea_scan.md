## Candidate review cards

### 1) AI Use Clarity Micro-Policy Card
- Lane: repo/free; Future of Education; paid later only for institution-specific rollout/review.
- Adaptation burden revealed: learners and workers are using AI before institutions translate policy into daily allowed/not-allowed behavior.
- Who carries it: students, employees, instructors, managers, HR/L&D, academic integrity staff, team leads.
- Missing layer: a one-task-at-a-time permission card that turns broad policy into operational rules for a specific assignment/workflow.
- Smallest proof artifact: Markdown card with task, allowed AI uses, disallowed uses, disclosure wording, source/evidence requirements, human-review trigger, and where to ask if unclear.
- Genericity-test risk: medium; many policies can be AI-generated, so value depends on being tiny, reusable, example-backed, and aligned with existing dossier/delegation tools.
- Safety/claim caveats: not legal/compliance advice; must not imply universal permissibility; local policy owner approval required before institutional use.
- Next build step: add three synthetic examples: college essay feedback, workplace meeting summary, nonprofit grant draft support.

### 2) AI Value Capture Evidence Log
- Lane: paid later; repo/free primitive first; monitor for institutional readiness pack.
- Adaptation burden revealed: organizations report widespread AI adoption but weak bottom-line/value evidence; teams are asked to prove ROI while workflows remain fragmented.
- Who carries it: team leads, operations managers, transformation offices, CFO/COO staff, frontline workers whose time is being reallocated.
- Missing layer: a humble evidence log that separates claimed productivity from observed workflow changes, quality effects, risk costs, and time shifted elsewhere.
- Smallest proof artifact: spreadsheet/Markdown schema: AI use case, baseline task time, post-change task time, review time, error/rework rate, human judgment saved/added, downstream bottleneck, evidence source, confidence.
- Genericity-test risk: low-medium if it forces before/after evidence and review costs; high if reduced to generic ROI claims.
- Safety/claim caveats: avoid productivity guarantee language; self-reported metrics are weak unless triangulated; do not use for worker surveillance without safeguards.
- Next build step: create a fictional 4-week team evidence log and a validator/checklist for “ROI claim not yet supported.”

### 3) Agent Identity & Scope Roster
- Lane: repo/free; paid later for implementation support; monitor security tooling market.
- Adaptation burden revealed: agents are becoming non-human actors with access, credentials, delegated authority, and unclear owners.
- Who carries it: IT/security, operations, tool owners, managers, compliance, anyone accountable for agent outputs.

- Smallest proof artifact: CSV/Markdown roster with agent name, owner, purpose, accounts/API keys, data classes, action permissions, spend/publish/write ability, expiry date, log location, review cadence.
- Genericity-test risk: low; concrete inventory fields and synthetic examples create value beyond a prompt.
- Safety/claim caveats: not a security control by itself; should not store secrets; avoid implying audit/certification.
- Next build step: extend Agent Permission Receipt / Toolchain Exposure Map with a roster example for a small team using three agents.
